This manual describes the features and customization options for the Emacs Lisp file =anki-noter.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=anki-noter.el= uses a large language model (via =gptel=) to generate Anki flashcards from various source materials and inserts them as org-mode headings formatted for =anki-editor=.  It bridges two workflows: reading source material and producing spaced-repetition cards, by delegating the extraction of atomic knowledge to an LLM.

The package accepts three kinds of input:

- The current buffer or active region.
- An external file, including PDFs (with native multimodal support for Anthropic and Google backends, and a =pdftotext= fallback for others).
- A URL, whose content is fetched and rendered before being sent to the LLM.

Generated cards are inserted as org-mode subtrees at point, with the correct heading level and property drawers (~ANKI_DECK~, ~ANKI_NOTE_TYPE~, ~ANKI_TAGS~, ~ANKI_FORMAT~) already in place.  After insertion, the package optionally pushes the new notes to Anki via =anki-editor=.

A template system lets users switch between prompt strategies for different domains—general non-fiction, language learning, programming—and custom templates can be added through the ~anki-noter-prompt-templates~ user option ([[#h:template-configuration][Template configuration]]).

The package requires Emacs 28.1 or later, =gptel= 0.9+, and =anki-editor= 0.3+.

* User options
:PROPERTIES:
:CUSTOM_ID: h:user-options
:END:

** Card generation
:PROPERTIES:
:CUSTOM_ID: h:card-generation-options
:END:

#+vindex: anki-noter-card-count
#+vindex: anki-noter-language
#+vindex: anki-noter-anki-format

The user option ~anki-noter-card-count~ sets the default target number of cards to generate.  It accepts either an integer or ~nil~.  When ~nil~ (the default), the LLM decides how many cards are appropriate for the given source material.  Set this to a specific number when you want consistent output size—for instance, when processing many short passages and you want roughly five cards from each.  This default can be overridden per invocation by calling any generation command with a prefix argument ([[#h:generating-cards][Generating cards]]).

The user option ~anki-noter-language~ controls the output language for generated cards.  When ~nil~ (the default), cards are generated in the same language as the source material.  Set this to a language name (e.g., ="Spanish"=) to produce cards in a different language—useful, for example, when reading English-language material but studying in your native language.

The user option ~anki-noter-anki-format~ specifies the value for the ~ANKI_FORMAT~ property on generated cards.  The default is ~nil~, which means cards use org-mode format.  Set this to a string if your =anki-editor= setup requires a specific format value.

** Template configuration
:PROPERTIES:
:CUSTOM_ID: h:template-configuration
:END:

#+vindex: anki-noter-default-template
#+vindex: anki-noter-prompt-templates

The user option ~anki-noter-default-template~ specifies which prompt template to use by default.  Its value must be a string matching a key in ~anki-noter-prompt-templates~.  The default is ="general"=.  Change this if you most often generate cards for a specific domain—for instance, set it to ="programming"= if you primarily read technical material.  You can also change it interactively during a session with ~anki-noter-select-template~ ([[#h:selecting-a-template][Selecting a template]]).

The user option ~anki-noter-prompt-templates~ is an association list of named prompt templates.  Each entry is a cons cell of ~(NAME . PROMPT-STRING)~, where ~NAME~ is a string used to identify the template and ~PROMPT-STRING~ is the domain-specific instruction appended to the base system prompt.  Three templates are provided by default:

- ="general"= — for non-fiction material: key facts, definitions, relationships, cause-effect pairs, and notable claims.
- ="language"= — for language learning: vocabulary, idiomatic expressions, grammar rules, and bilingual cards when the output language differs from the source.
- ="programming"= — for programming and computer science: concepts, API facts, code behavior, algorithm properties, and design patterns, with an emphasis on understanding over recall.

To add a custom template, push a new cons cell onto this list:

#+begin_src emacs-lisp
(add-to-list 'anki-noter-prompt-templates
             '("philosophy" . "You are generating flashcards about philosophy. Focus on arguments, distinctions, thought experiments, and key claims by specific thinkers."))
#+end_src

** Citation
:PROPERTIES:
:CUSTOM_ID: h:citation-options
:END:

#+vindex: anki-noter-cite-format

The user option ~anki-noter-cite-format~ controls the format string used for source citations appended to the back of each card.  The =%s= placeholder is replaced with the source description (file name, URL, or buffer name).  The default is ="Source: %s"=.  Customize this if you prefer a different citation style—for example, ="Ref: %s"= or ="[%s]"=.

** PDF handling
:PROPERTIES:
:CUSTOM_ID: h:pdf-handling-options
:END:

#+vindex: anki-noter-pdf-fallback-command

The user option ~anki-noter-pdf-fallback-command~ specifies the shell command used to extract text from PDF files when the current =gptel= backend does not support native PDF input.  The =%s= placeholder is replaced with the (shell-quoted) file path.  The default is ="pdftotext %s -"=, which sends the extracted text to standard output.

Native PDF support is detected automatically: Anthropic (Claude) and Google (Gemini) backends can process PDF files directly via ~gptel-context-add-file~, so this fallback is only used with other backends such as OpenAI.  If =pdftotext= is not available on your system, install it (it is part of the =poppler-utils= package on most systems) or configure this option to use an alternative tool.

** Anki integration
:PROPERTIES:
:CUSTOM_ID: h:anki-integration-options
:END:

#+vindex: anki-noter-auto-push

The user option ~anki-noter-auto-push~ controls whether newly generated notes are pushed to Anki automatically.  When ~nil~ (the default), the package prompts you with a confirmation after inserting cards.  Set this to ~t~ to skip the prompt and push immediately—convenient when you trust the LLM output and want a streamlined workflow.  Pushing requires =anki-editor= and a running AnkiConnect instance.

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Generating cards
:PROPERTIES:
:CUSTOM_ID: h:generating-cards
:END:

#+findex: anki-noter-generate
#+findex: anki-noter-generate-from-file
#+findex: anki-noter-generate-from-url

All three generation commands share the same core workflow: they determine the insertion point and heading level from the current org-mode context, build a system prompt from the active template and options, send the source material to the LLM via =gptel=, parse the response, and insert the resulting org headings at point.  After insertion, they optionally push the new notes to Anki ([[#h:anki-integration-options][Anki integration]]).

The commands also perform /incremental generation/: they scan sibling headings that already have an ~ANKI_NOTE_TYPE~ property and include their front text in the prompt, instructing the LLM not to produce duplicate cards.  This means you can run a generation command multiple times on the same source and get fresh cards each time.

All three commands accept a prefix argument.  Without a prefix argument, they use the defaults from ~anki-noter-default-template~ and ~anki-noter-card-count~ ([[#h:card-generation-options][Card generation]]).  With a prefix argument (=C-u=), they interactively prompt for:

- *Template* — selected via ~completing-read~ from the keys of ~anki-noter-prompt-templates~.
- *Card count* — an integer, or empty to let the LLM decide.
- *Topic focus* — a string to narrow generation to a specific topic, or empty for no filter.

If the org context does not already provide ~ANKI_DECK~ and ~ANKI_TAGS~ properties (via inheritance up the heading tree), the commands prompt for these values interactively.

The command ~anki-noter-generate~ generates cards from the current buffer.  If a region is active, only the selected text is sent to the LLM; otherwise, the entire buffer content is used.  Call this command with =M-x anki-noter-generate= while visiting the source material in an org-mode buffer.

The command ~anki-noter-generate-from-file~ generates cards from an external file.  It prompts for a file path and handles three cases transparently: PDF files on backends with native PDF support (Anthropic, Google) are added to the =gptel= context directly; PDF files on other backends are processed through the fallback command ([[#h:pdf-handling-options][PDF handling]]); and text-based files are added to the =gptel= context.  For PDF files, it also prompts for an optional page range (e.g., ="1-20"=).

The command ~anki-noter-generate-from-url~ generates cards from a web page.  It prompts for a URL, fetches the page asynchronously, renders the HTML to plain text using ~shr-render-region~, and then sends the extracted text to the LLM.  The URL itself is used as the source citation.

** Selecting a template
:PROPERTIES:
:CUSTOM_ID: h:selecting-a-template
:END:

#+findex: anki-noter-select-template

The command ~anki-noter-select-template~ lets you change the active prompt template for the current session.  It presents the available template names from ~anki-noter-prompt-templates~ via ~completing-read~ and sets ~anki-noter-default-template~ to the chosen value.  This is a quick way to switch domains without customizing the variable permanently—for instance, switching from ="general"= to ="language"= when you move from reading a textbook to studying vocabulary.

** Prompt assembly
:PROPERTIES:
:CUSTOM_ID: h:prompt-assembly
:END:

#+findex: anki-noter-prompts-build

The function ~anki-noter-prompts-build~ assembles the full system prompt sent to the LLM.  It takes a template name, deck string, and tags string as positional arguments, plus the following keyword arguments: ~:card-count~, ~:language~, ~:cite-source~, ~:existing-cards~, and ~:topic~.

The function retrieves the template text from ~anki-noter-prompt-templates~, constructs the base prompt (which includes the output format specification, card quality guidelines, and any conditional sections for citation, card count, language, topic focus, and existing cards), and appends the template-specific instructions.

This function is useful if you want to build prompts programmatically for custom workflows—for example, to generate cards in batch or to inspect the prompt before sending it:

#+begin_src emacs-lisp
(let ((prompt (anki-noter-prompts-build
               "general" "Default" "study"
               :card-count 10
               :language "Spanish"
               :cite-source "my-notes.org")))
  (message "%s" prompt))
#+end_src
